## 1. 背景・目的

### 1.1 背景
個人投資（中長期・低回転）における銘柄選定を、再現性ある定量プロセスで支援するツールを構築する。既存ポートフォリオ（SP500 60万円 / 高配当日本株 20万円）に対して、相関の低い資産を加えて分散を効かせることを狙う。

### 1.2 目的
- ルールベース＋定量スコアリングで候補銘柄を再現性をもって抽出する
- LLM（Bedrock/Claude）は「選定理由の説明生成」のみに使用し、数値判断には用いない
- AWS最小構成で運用コストを抑え、無料枠＋月数百円以内で運用する

### 1.3 設計哲学（重要）
- **AIに数値判断・計算をさせない**：ハルシネーション・再現性喪失を防ぐ
- **選定の核はコード、LLMは翻訳者**：retrievalに決算/IR、generationに説明文
- **最終購入判断は人間**：システムは候補提示のみで発注しない

---

## 2. スコープ

### 2.1 Phase1（MVP）
- 日本株のみ対象（J-Quants）
- 週次バッチで候補銘柄抽出 → Bedrockでレポート生成 → Google Drive配置 → メール通知
- 提示履歴のDB記録

### 2.2 Phase2（拡張、本ドキュメントでは設計のみ考慮）
- 米国株（SEC EDGAR）、マクロ指標（FRED）、為替（USD/JPY）対応
- ETF・REIT・債券・金などの資産クラス追加
- ユーザの投資結果INPUT（実購入記録の反映）
- 既存ポートフォリオとの相関評価高度化

### 2.3 対象外（恒久的）
- 売買発注の自動化（最終判断は人間）
- 短期トレーディング・テクニカル分析中心の評価
- 外部公開Webサイト・他ユーザ向け機能

## 4. 機能要件

| ID | 機能 | 概要 | Phase |
|---|---|---|---|
| F-01 | 市場データ収集 | J-Quants APIから株価・財務・配当データを取得しDynamoDBに保管 | 1 |
| F-02 | スクリーニング | 単元購入額・配当利回り・自己資本比率・PER・流動性などで粗フィルタ。**全閾値を設定ファイルでパラメータ化** | 1 |
| F-03 | スコアリング | 配当継続性・割安度・財務健全性・既存PFとの相関を重み付け合成。**評価軸・重みを設定ファイルでパラメータ化** | 1 |

| F-07 | 通知 | レポート生成完了をSNS経由でメール通知 | 1 |
| F-08 | 提示履歴記録 | 「いつ・何を・どのスコアで提示したか」をDynamoDBに記録 | 1 |
| F-09 | 米国株対応 | SEC EDGARからのデータ取得・評価 | 2 |
| F-10 | マクロ指標 | FRED（金利・インフレ）取得・局面判断 | 2 |
| F-11 | 為替対応 | USD/JPY取得、既存SP500の円換算評価 | 2 |
| F-12 | 資産クラス拡張 | ETF/REIT/債券/金など資産クラス別スコアリング | 2 |
| F-13 | 実購入記録INPUT | ユーザの実購入を記録、後続のリバランス評価に利用 | 2 |

## 5. 非機能要件

| 分類 | 要件 |
|---|---|
| コスト | AWS無料枠 ＋ Bedrock利用料を含めて月100円未満を目標、上限月500円 |
| 可用性 | 週次バッチで処理完了すればよい。SLA・SLOは緩い |
| 性能 | 1回のバッチ実行が30分以内に完了 |
| 再現性 | 各ステージは冪等で再実行可能。実行ログを残す |
| セキュリティ | 外部公開なし。認証情報はAWS Secrets Managerで管理。Lambdaは最小IAM権限 |
| 拡張性 | データソース・資産クラス・スコア評価軸を追加しやすい構造（アダプタパターン・設定ファイル化） |
| 運用性 | スクリーニング閾値・スコア重みの変更はLambda再デプロイなしで反映可能 |
| データ保全 | DynamoDBはPITR有効化、S3はバージョニング有効化 |

## 6. 制約・前提

| 項目 | 値 |
|---|---|
| 月次予算 | 5〜10万円 |
| 1銘柄あたり上限 | 10万円未満（株価1,000円未満の日本株が中心） |
| 売買頻度 | 低回転（あまり売らない） |
| 投資期間 | 中長期 |
| 既存ポートフォリオ | SP500 60万円 / 高配当日本株 20万円 |
| NISA枠 | 要確認（スコア計算に税引後リターンとして織り込む想定） |

## 7. パラメータ化方針（重要）

スクリーニング条件・スコア重み・閾値は**全て設定ファイル（YAML）で外出し**し、コード変更なしで調整できるようにする。

### 7.1 パラメータの所在
- S3バケットに `config.yaml` を配置
- Lambda起動時にS3から読み込み
- バージョニング有効化により過去設定にロールバック可能
- Lambda起動時にYAMLスキーマ検証を実施し、不正値はエラーで停止

### 7.2 スクリーニング条件（初期値・後から変更可）
| パラメータ | 初期値 | 意味 |
|---|---|---|
| max_unit_price_jpy | 100,000 | 単元購入額上限（円） |
| min_dividend_yield | 0.030 | 配当利回り下限（3.0%） |
| min_equity_ratio | 0.40 | 自己資本比率下限（40%） |
| max_per | 20 | PER上限（倍） |
| min_daily_volume | 1,000 | 1日平均出来高下限（株） |
| min_continuous_dividend_years | 5 | 連続増配年数下限（任意、無効化可） |

### 7.3 スコア重み（初期値・合計1.0）
| 評価軸 | 重み | 意味 |
|---|---|---|
| dividend_continuity | 0.35 | 配当継続性（連続増配年数・配当性向） |
| valuation | 0.30 | 割安度（PER・PBR） |
| financial_health | 0.20 | 財務健全性（自己資本比率・有利子負債） |
| portfolio_correlation | 0.15 | 既存PFとの相関（低いほど高評価） |

### 7.4 候補生成パラメータ
| パラメータ | 初期値 | 意味 |
|---|---|---|
| top_n | 10 | 上位N銘柄を候補とする |
| monthly_budget_jpy | 100,000 | 月次予算（候補の組合せ最適化に使用） |

## 8. データソース

| 分類 | データ | ソース | Phase | 用途 |
|---|---|---|---|---|
| 市場 | 株価・出来高 | J-Quants | 1 | 価格上限フィルタ・ボラティリティ |
| 財務 | PER/PBR/配当利回り/財務諸表 | J-Quants | 1 | スクリーニング・スコア中核 |
| 配当 | 配当性向・連続増配年数 | J-Quants/TDnet | 1 | 中長期×高配当戦略 |
| 定性 | 決算・適時開示 | TDnet | 1 | Bedrock RAGで説明生成 |
| 自己情報 | 現保有・取得単価 | 設定ファイル（Phase1は静的） | 1→2で動的化 | 相関・重複評価 |
| 市場 | 米国株株価・財務 | SEC EDGAR | 2 | 米国株評価 |
| マクロ | 金利・インフレ・雇用 | FRED | 2 | 局面判断・配分 |
| 為替 | USD/JPY | FRED | 2 | 円換算評価 |

## 9. 落とし穴回避策

| 落とし穴 | 対策 |
|---|---|
| ルックアヘッドバイアス | 決算発表"日"でデータを区切る。財務データに `as_of_date` を必ず持たせる |
| 過剰適合 | 検証期間を分ける（in/out-of-sample）。重みは設定ファイルで管理し変更履歴を残す |
| LLMの数値ハルシネーション | LLMには計算をさせず、コード由来の数値を"引用"のみさせる |
| 為替の無視 | Phase2でUSD/JPYを必ず取得し、米国株は円換算評価 |
| 手数料・税の無視 | スコアに税引後リターン換算を組み込み（NISA枠考慮） |
| データ欠損 | 欠損フラグと再取得ロジックをIngestion層に組み込む |
| 既存PFとの重複 | 既存PFとの相関をスコア軸に組み込み（重み 0.15） |

## 10. 用語

| 用語 | 説明 |
|---|---|
| 単元 | 日本株の最小売買単位（多くは100株） |
| 単元購入額 | 株価 × 100株（単元購入に必要な金額） |
| ファンダメンタル | 財務諸表・配当履歴など企業の基礎的指標 |
| RAG | Retrieval-Augmented Generation。検索結果をLLMに渡して応答生成 |
| ルックアヘッドバイアス | バックテストで未来情報を使ってしまう誤り |
