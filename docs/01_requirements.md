## 1. 背景・目的

### 1.1 背景
個人投資（中長期・低回転）における銘柄選定を、再現性ある定量プロセスで支援するツールを構築する。既存ポートフォリオ（SP500 60万円 / 高配当日本株 20万円）はまだ完成ではなく、SP500・高配当日本株の買い増しも含め、**現状の資産を維持・成長させること**を目的とする。資産クラスによる恣意的な除外はしない。

### 1.2 目的
- ルールベース＋定量スコアリングで候補銘柄を再現性をもって抽出する
- LLM（Bedrock/Claude）は「選定理由の説明生成」のみに使用し、数値判断には用いない
- AWS最小構成で運用コストを抑える（AWSコストは月100円目標）
- データ取得コスト（J-Quants）はAWSコストとは別枠。本番運用はLight（月1,650円）が前提

### 1.3 設計哲学（重要）
- **AIに数値判断・計算をさせない**：ハルシネーション・再現性喪失を防ぐ
- **選定の核はコード、LLMは翻訳者**：retrievalに決算/IR、generationに説明文
- **最終購入判断は人間**：システムは候補提示のみで発注しない

## 2. スコープ

### 2.1 Phase1（MVP）
- 日本株のみ対象（J-Quants）
- 週次バッチで候補銘柄抽出 → Bedrockでレポート生成 → Google Drive配置 → メール通知
- 提示履歴のDB記録

### 2.2 Phase2（拡張、本ドキュメントでは設計のみ考慮）
- 米国株（SEC EDGAR）、マクロ指標（FRED）、為替（USD/JPY）対応
- ETF・REIT・債券・金などの資産クラス追加
- ユーザの投資結果INPUT（実購入記録の反映）
- 既存保有の動的管理（実購入履歴の反映、同一銘柄の重複チェック強化）

### 2.3 対象外（恒久的）
- 売買発注の自動化（最終判断は人間）
- 短期トレーディング・テクニカル分析中心の評価
- 外部公開Webサイト・他ユーザ向け機能

## 5. 非機能要件

| 分類 | 要件 |
|---|---|
| コスト（AWS） | Bedrock利用料を含めて月100円未満を目標、上限月500円 |
| コスト（データ） | J-Quants Light 月1,650円（本番前提）。開発中はFree（12週遅延）。Standard（3,300円）は連続増配10年判定が必要になった段階で検討 |
| コスト（合計） | 月1,800〜2,000円程度を許容コストとして設計する。月5〜10万投資前提で許容範囲 |
| 可用性 | 週次バッチで処理完了すればよい。SLA・SLOは緩い |
| 性能 | 1回のバッチ実行が30分以内に完了 |
| 再現性 | 各ステージは冪等で再実行可能。実行ログを残す |
| セキュリティ | 外部公開なし。認証情報はAWS Secrets Managerで管理。Lambdaは最小IAM権限 |
| 拡張性 | データソース・資産クラス・スコア評価軸を追加しやすい構造（アダプタパターン・設定ファイル化） |
| 運用性 | スクリーニング閾値・スコア重みの変更はLambda再デプロイなしで反映可能 |
| データ保全 | DynamoDBはPITR有効化、S3はバージョニング有効化 |

## 7. J-Quantsプラン前提

### 7.1 APIバージョン方針

**本システムはJ-Quants V2 APIのみを使用する。** V1（旧方式：mail/password/refresh_token認証）は実装しない。2025年12月以降の新規登録はV2（APIキー方式）のみ提供されており、V2を前提とすることで将来的な非互換リスクを排除する。

### 7.2 プラン選定と方針

**結論：本番運用にはLight（月1,650円）以上が必須。** 以下の2つの理由からFreeは本番利用不可とする。

**理由① CSV一括取得が本番初期ロードに必要**

初回起動時に過去5年分の株価ヒストリカル（PriceHistory）を一括投入する必要がある。この件数は「4,000銘柄 × 252営業日 × 5年 ≒ 500万レコード超」になる。REST APIのみで取得しようとすると、Lightプランの上限60件/分でも数日かかり現実的でない。**J-QuantsのCSV一括取得機能（過去データを日付範囲で一括ダウンロード）を使うことで、初期ロードを数時間以内に完了できる。FreeプランはCSV一括取得が利用不可のため、初期ロードの時点で詰まる。**

初期ロード完了後の毎週のデータ追加（前週分のみ）は通常のREST API（V2）を使う。

**理由② 12週遅延でスクリーニングが機能しない**

Freeプランは取得できるデータが12週（約3ヶ月）遅延する。週次バッチで最新の株価・財務情報を使ってスクリーニングすることが本システムの目的であり、3ヶ月遅れのデータでは意味をなさない。

| フェーズ | プラン | 月額 | 採用理由 |
|---|---|---|---|
| 開発・検証 | Free | 0円 | API仕様確認・フィールド構造の把握・コード動作確認のみ。本番には使わない |
| **本番運用（推奨）** | **Light** | **1,650円** | **CSV一括取得可・最新データ・5年分・60件/分**。本番に必要な条件を全て満たす最低プラン |
| 本番運用（精度向上） | Standard | 3,300円 | 10年分で連続増配・財務トレンドの判定精度向上。Lightと同じコードで動作 |
| — | Premium | 16,500円 | 詳細財務三表・配当金詳細が必要になるまで不要 |

**移行戦略**：Free（開発・API確認）→ **Light（本番開始・初期ロード実施）** → 必要になったらStandard。

### 7.3 取得可能なデータの上限（Phase1設計への影響）
| 設計書の想定 | 実際の取得可能範囲 | 対応方針 |
|---|---|---|
| 財務諸表（詳細BS/PL/CF） | **Premium限定** | Phase1は財務サマリーで代替。PER/PBR/配当利回り/自己資本比率はサマリーから算出可 |
| 配当金情報（権利日・実績配当額） | **Premium限定** | 配当予想・配当性向は財務サマリーで取得可。詳細はPhase2以降 |
| TDnet適時開示テキスト | アドオン+11,000円/月 | 費用対効果が低いため不採用。**EDINET API（無料）**で有価証券報告書・決算短信を代替 |
| 連続増配年数 | Light=5年分、Standard=10年分 | Light採用時は「5年以上連続増配」が最大判定範囲。スクリーニング閾値はこれに合わせて設定 |

### 7.4 規約上の重要事項（必読）
- **データ配布禁止**：取得データそのものの第三者配布は規約違反
- **分析結果の継続配信禁止**：副業サービスとして銘柄分析結果を顧客に継続提供する行為は規約抵触の可能性あり
- **コード公開はOK**：GitHubへのコード（データを含まない）公開は問題なし
- **解約後のデータ廃棄**：解約時にDB上の取得データを消去する義務あり
- **V2 API必須**：2025年12月以降の新規登録はAPIキー方式（V2）のみ

> 上記は設計時点の理解。利用前に必ず[J-Quants利用規約](https://jpx-jquants.com/)の原文を確認すること。

## 8. パラメータ化方針（重要）

スクリーニング条件・スコア重み・閾値は**全て設定ファイル（YAML）で外出し**し、コード変更なしで調整できるようにする。

### 8.1 パラメータの所在
- S3バケットに `config.yaml` を配置
- Lambda起動時にS3から読み込み
- バージョニング有効化により過去設定にロールバック可能
- Lambda起動時にYAMLスキーマ検証を実施し、不正値はエラーで停止

### 8.2 スクリーニング条件（初期値・後から変更可）
| パラメータ | 初期値 | 意味 |
|---|---|---|
| max_unit_price_jpy | 100,000 | 単元購入額上限（円） |
| min_dividend_yield | 0.030 | 配当利回り下限（3.0%） |
| min_equity_ratio | 0.40 | 自己資本比率下限（40%） |
| max_per | 20 | PER上限（倍） |
| min_daily_volume | 1,000 | 1日平均出来高下限（株） |
| min_continuous_dividend_years | 5 | 連続増配年数下限（任意、無効化可） |
| exclude_existing_tickers | true | 既に保有している同一銘柄を候補から除外（同一銘柄の重複買い防止のみ。資産クラス・相場との相関は問わない） |

### 8.3 スコア重み（初期値・合計1.0）
| 評価軸 | 重み | 意味 |
|---|---|---|
| dividend_continuity | 0.20 | 配当継続性（連続増配年数・配当性向） |
| valuation | 0.30 | 割安度（PER・PBR） |
| financial_health | 0.50 | 財務健全性（自己資本比率・有利子負債） |

> **Note**: 既存PFとの相関はスコアリング軸から除外。SP500・高配当日本株との相関が高くてもスコアは下がらない。同一銘柄の重複は §8.2 スクリーニング条件の `exclude_existing_tickers` フラグで制御する。

### 8.4 候補生成パラメータ
| パラメータ | 初期値 | 意味 |
|---|---|---|
| top_n | 10 | 上位N銘柄を候補とする |
| monthly_budget_jpy | 100,000 | 月次予算（候補の組合せ最適化に使用） |

| マクロ | 金利・インフレ・雇用 | FRED | 2 | 局面判断・配分 |
| 為替 | USD/JPY | FRED | 2 | 円換算評価 |

## 10. 落とし穴回避策

| 落とし穴 | 対策 |
|---|---|
| ルックアヘッドバイアス | 決算発表"日"でデータを区切る。財務データに `as_of_date` を必ず持たせる |
| 過剰適合 | 検証期間を分ける（in/out-of-sample）。重みは設定ファイルで管理し変更履歴を残す |
| LLMの数値ハルシネーション | LLMには計算をさせず、コード由来の数値を"引用"のみさせる |
| 為替の無視 | Phase2でUSD/JPYを必ず取得し、米国株は円換算評価 |
| 手数料・税の無視 | スコアに税引後リターン換算を組み込み（NISA枠考慮） |
| データ欠損 | 欠損フラグと再取得ロジックをIngestion層に組み込む |
| 既存銘柄の重複買い | `exclude_existing_tickers: true` でスクリーニング時に同一銘柄を除外。ただし資産クラス・相場との相関は除外条件にしない（SP500・高配当日本株の追加購入は常に選択肢） |
| **J-Quants規約違反** | **コードのGitHub公開はOK。ただしJ-Quantsで取得したデータそのものの配布・再配信は禁止。「分析結果」を第三者に継続反復提供する行為（例：副業サービスとして銘柄分析を顧客に配信）も規約抵触の可能性あり。用途は個人利用に限定すること。** 規約原文を必ず確認すること |
| J-Quants解約時のデータ廃棄 | 規約により解約後は取得データの消去義務あり。DynamoDBのデータは解約後に使用不可となる前提で設計すること |
| J-Quants API バージョン | 2025年12月以降の新規登録はV2 API（APIキー方式）のみ。実装はV2前提とし、旧方式（mail/password/refresh_token）では組まないこと |

## 11. 用語

| 用語 | 説明 |
|---|---|
| 単元 | 日本株の最小売買単位（多くは100株） |
| 単元購入額 | 株価 × 100株（単元購入に必要な金額） |

| ルックアヘッドバイアス | バックテストで未来情報を使ってしまう誤り |
