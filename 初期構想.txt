# 自分用の投資支援ツールの作成方針


## ① 結論（設計の核）

- **AIの役割を「選定そのもの」に置かない。** 選定の核は**ルールベース＋定量スコアリング**で再現性を担保し、LLMは「定性情報の構造化」と「なぜこの銘柄かの説明生成」に限定する。
- LLMに数値判断や計算をさせると、ハルシネーションと再現性喪失で破綻する。ここが最大の設計分岐点。
- 全体は **データ収集 → 蓄積 → スクリーニング → スコアリング → 候補生成 → 説明生成 → 提示 → 記録/モニタリング** のパイプライン。エージェント的な自律ループは中長期・低回転には不要。

---

## ② 前提・スコープの明文化（パラメータ化すべき制約）

システムを作る前に、これらを設定ファイル（YAML/JSON）として外出しします。SREの感覚でいう「環境変数・パラメータストア」に当たる部分です。

| 項目 | 値（なべさんの想定） | システム上の意味 |
|---|---|---|
| 月次予算 | 5〜10万円 | 候補の価格上限フィルタ |
| 1銘柄あたり上限 | 10万円未満 | 「100株単位で10万未満」＝株価1,000円未満の日本株、または米国株・ETF（1株単位）が候補になりやすい |
| 売買頻度 | 低回転（あまり売らない） | 短期シグナルを評価軸から外す |
| 投資期間 | 中長期 | テクニカルより**ファンダメンタル＋配当継続性**を重視 |
| 既存ポートフォリオ | SP500 60万 / 高配当日本株 20万 | **相関の低い資産**を足す＝分散の評価軸が必要 |
| 対象資産 | 株式＋その他 | ETF・REIT・金などを同じ枠組みで評価 |
| 税制 | （要確認）NISA枠の有無 | 手数料・税引後リターンに直結。スコアに織り込む |

ポイントは、**既存ポートフォリオがSP500に大きく偏っている**こと。新規購入の評価軸に「既存資産との相関」を入れないと、似たものを買い増して分散が効かなくなります。

---

## ③ 必要なINPUT（データソースの分類）

「一覧→分類」で整理します。料金・提供条件は変わりやすいので**要確認**としてください。

| 分類 | 具体データ | データソース例（要確認） | 用途 |
|---|---|---|---|
| **市場データ** | 株価・出来高（ヒストリカル/日次） | J-Quants（日本株・無料枠あり）, yfinance（手軽だが非公式）, Alpha Vantage | 価格上限フィルタ、ボラティリティ算出 |
| **為替** | USD/JPY | FRED, exchangerate系API | SP500保有の円換算・新規USD投資の判断 |
| **ファンダメンタル** | PER/PBR/配当利回り/財務諸表/増配履歴 | J-Quants, SEC EDGAR（米国・無料） | スクリーニングとスコアの中核 |
| **配当・株主還元** | 配当性向・連続増配年数 | 適時開示（TDnet）, 各社IR | 中長期×高配当戦略の核 |
| **定性/ニュース** | 決算・適時開示・ニュース | TDnet, ニュースAPI | LLMで要約・センチメント構造化 |
| **マクロ** | 金利・インフレ・雇用 | FRED（無料） | 局面判断・債券/金の配分 |
| **自分の状態** | 現保有・取得単価・配分 | 自前DB（手入力 or 証券CSV） | 相関・重複・リバランス判定 |
| **制約/選好** | 予算・上限・リスク許容 | 設定ファイル | 全プロセスの制御パラメータ |

無料で始めるなら **J-Quants（日本株）＋ SEC EDGAR（米国）＋ FRED（マクロ）＋ yfinance（価格）** の組み合わせがコスパ良いです。

## ④ プロセス構成（パイプライン）

線形パイプラインとして、各段の責務を分離します。SREの目で見れば「各ステージは冪等で、再実行可能、ログが残る」設計が理想です。

```
[1] Ingestion（収集）
     └ 各APIから日次バッチで取得。失敗時リトライ、欠損フラグ。
         ↓
[2] Storage（蓄積・正規化）
     └ DB（SQLite/Postgres）に正規化保存。価格・財務・自分の保有を分離テーブルで。
         ↓
[3] Screening（ルールベース・粗フィルタ）  ← ここはLLMを使わない
     └ 例：単元購入額 < 10万 / 配当利回り > X% / 自己資本比率 > Y% / 流動性下限
         ↓
[4] Scoring（定量スコアリング）           ← ここもLLMを使わない
     └ ファンダ・配当継続性・割安度・既存PFとの相関 を重み付けで合成 → 総合スコア
         ↓
[5] Ranking & 候補生成
     └ 上位N銘柄を抽出。予算内に収まる組み合わせを提示。
         ↓
[6] Explanation（説明生成）               ← ここでLLM/RAGを使う
     └ 決算・ニュースをRAGで引き、「なぜ候補に挙がったか」を自然言語化。
         数値はスコアリング結果を“引用”させ、LLMに計算させない。
         ↓
[7] Presentation（提示）
     └ なべさんが読んで最終判断・購入（システムは発注しない）。
         ↓
[8] Logging & Monitoring（記録・追跡）
     └ 「いつ・何を・どのスコアで提示し、買ったか」を記録。
         買った後の値動き・配当を追跡し、後で振り返れる状態に。
         ↓
   （フィードバック）スコアの重みやフィルタ閾値を見直す
```

**設計上の肝は [3][4] と [6] の分離です。** 選定の根拠（数値）は決定論的なコードで出し、LLMはその根拠を「説明する翻訳者」として使う。これでハルシネーションが判断に混入しません。なべさんが学んできたRAGの構図そのまま、「retrieval＝決算/IR情報」「generation＝説明文」で当てはまります。

## ⑤ AIエージェント／LLMの役割の切り分け（重要な決定指針）

| タスク | 担当 | 理由 |
|---|---|---|
| 価格・財務の取得 | コード | 正確性・再現性 |
| フィルタ・スコア計算 | コード | LLMに数値を任せない |
| 決算/ニュースの要約・論点抽出 | LLM | 非構造データの構造化が得意 |
| センチメント分類 | LLM（or 軽量MLモデル） | 定性判断 |
| 「なぜこの銘柄か」の説明文生成 | LLM | 自然言語化が本領 |
| 最終購入判断 | **人間（なべさん）** | 責任とリスク許容は人間が持つ |

エージェント（ReAct的な自律ツール選択）を入れるなら、**[1]の収集オーケストレーション**や、**[6]で「どのIR資料を引くか」をLLMに選ばせる**あたりが適所。売買判断ループにはしない。

## ⑥ 株式以外の組み込み方

同じスコアリング枠組みに「資産クラス」属性を足すだけで拡張できます。10万未満・中長期・低回転の制約に合いやすいのは：

| 資産 | 組み込み方 | 注意 |
|---|---|---|
| ETF（米/日） | 個別株と同じ枠で評価。1株単位なので10万未満が容易 | 既存SP500との重複に注意 |
| REIT | 配当利回り軸でスコア。日本株高配当と相関を見る | 金利感応度が高い |
| 債券/債券ETF | マクロ（金利）データと連動評価 | 株と逆相関でヘッジ役 |
| 金（金ETF/積立） | インフレ・実質金利を入力に | 配当ゼロなのでスコア軸を分ける |
| 暗号資産 | 別枠・少額・高ボラ前提 | 評価ロジックを株と混ぜない |

「資産クラスごとにスコアの評価軸を切り替える」設計（株は配当＋割安、金はマクロ、債券は金利）にしておくと拡張が破綻しません。

## ⑦ よくある落とし穴

- **ルックアヘッドバイアス**：バックテストで「その時点では知り得ない未来データ」を使ってしまう。決算発表“日”でデータを区切る。
- **過剰適合**：過去にだけ効くパラメータを掘り当てて満足する。検証期間を分ける（in/out-of-sample）。
- **LLMの説明が説得的＝正しい、の錯覚**：流暢な根拠ほど疑う。数値はコード由来か必ず照合。
- **為替の無視**：SP500保有者にとってUSD/JPYはリターンの大きな部分。円換算で評価する。
- **手数料・税の無視**：税引後・手数料後で評価しないと中長期で効いてくる。NISA枠の扱いを最初に決める。
- **データ欠損の放置**：APIの欠損・遅延がスコアを歪める。欠損フラグと再取得を設計に入れる。
- **既存PFとの重複**：SP500偏重なのに似た米国大型を足し続けると分散が効かない。相関を必ず評価軸に。

## ⑧ 次のアクション（MVPの作り方）

最初から全部作らず、**最小で回るパイプライン**から始めるのがおすすめです。

1. **[2]+[3] の最小版**：J-Quantsで日本株の「株価＋配当利回り＋単元10万未満」だけフィルタするスクリプトをまず作る（LLMなし）。
2. **[4] スコアの初版**：配当利回り・自己資本比率・PERの3軸を重み付け合成。重みは設定ファイルへ。
3. **[8] 記録**：提示した候補をCSV/DBに残すだけ。後で振り返れる状態を先に作る。
4. **[6] 説明生成を後付け**：候補が出るようになってから、決算サマリをLLMで付ける。
5. 慣れたら米国株（EDGAR）・ETF・為替を足し、資産クラス軸を拡張。

GitHub公開も視野に入れているとのことなので、`config.yaml`（制約）／`ingest/`／`screen/`／`score/`／`explain/`とモジュールを分けておくと、後で公開・共有しやすい構成になります。
